1. Context Diagram

Purpose:

This diagram shows the overall context of the system, external actors, and how data flows between them. While similar to other diagrams on our diagram section, it is clearer and much more simplified.

Explanation:

The Context Diagram shows how the fall detection system interacts with external actors. It highlights the user with the mobile app, responders, system administrators, emergency services, and external APIs like Google Maps. At the center is the backend server, which manages all system functionality and communication. This diagram provides a high-level overview of the system's environment, showing who uses it and what services it connects to.

Actor / Component Description
User (Mobile App) A person using the fall detection app; their device detects motion and triggers fall events.
Medical Responder A nearby trained individual who receives alerts and responds to fall incidents.
Backend Server Central cloud-based server that handles fall event logic, data storage, notifications, and APIs.
Admin Console Internal dashboard used by system administrators to manage users, events, and logs.
Emergency Services External public safety services (e.g., 911) contacted if no responder accepts the alert.
Google Maps / APIs External services used to get GPS, routing, geolocation, and AED locations.

2. Container Diagram

Purpose:

Breaks down the system into major technology components to ensure readability and an easy understanding of the components.

Explanation:

The Container Diagram breaks down the system into its major building blocks, or containers, such as the user's mobile app, the backend server, the responder app, and the admin console. It shows how each container is responsible for specific tasks (e.g., fall detection, notifications, user management) and how they communicate via APIs and a shared database. This diagram provides a software architecture view of the system's deployment and responsibilities.

Component Description
User Mobile App Flutter/Kotlin/Swift app used by the user to interact with sensors and report events.
Web Backend Node.js, Django, or Flask server handling REST API, user data, and event logic.
Responder App (Web/PWA) Frontend interface used by responders to receive, view, and respond to alerts.
Admin Console Web-based internal tool used to monitor, configure, and log system activity.
Databases PostgreSQL/MongoDB used to store users, events, AED locations, and logs.
External APIs Google Maps API, location services, and potentially SMS gateways for alerts.

3. Component Diagram

Purpose:

Goes into more detail about each component to show how each section works.

Explanation:

The Component Diagram focuses on the internal structure of the backend server. It illustrates how modules like the Fall Event Processor, Notification Service, User Profile Service, and AED Locator work together to handle incoming events, notify responders, and manage user data. This diagram helps clarify the specific services and logical components within the backend, making it easier to understand how the system processes fall events.

Actors (within Backend Processing)

Actor / Component Description
Fall Detection Processor Receives fall events and manages responder assignment and fallback logic.
User Profile Service Stores user data, medical profiles, DNR info, and makes them available to responders or EMS.
Responder Notification Module Finds and alerts the nearest responder(s) based on geolocation.
AED Locator Provides nearest AED location data based on event coordinates.
Emergency Escalation Handler Sends fallback alert to emergency services if no responder accepts in time.
Admin Module Handles admin permissions, user roles, and dashboard access.

Components Description (Backend Modules & Data Stores)

Component Description
REST API Entry point for all client requests (fall events, user data, responder actions).
Fall Event Processor Main controller that handles incoming fall events and coordinates the response logic.
Notification Service Finds and notifies nearby responders via push/SMS, tracks response timeouts.
User Profile Service Retrieves user details, medical data, and DNR preferences for responders and EMS.
AED Locator Searches for nearby AEDs based on event location using geo-tagged data.
Emergency Escalation Handler Triggers an alert to emergency services if no responder accepts within a time limit.
Responder App Interface used by responders to receive alerts and view case details.
User DB Stores user accounts, medical data, responder roles, and event history.
AED DB Contains location data for all registered AEDs in the system.

4. Sequence Diagram (Fall Event)

Purpose:

Visualize what occurs when someone falls while using the "Fall Detection" application.

Explanation:

The Sequence Diagram shows the step-by-step flow of a fall event from detection to response. It begins with motion being detected on the user's device, followed by a confirmation prompt. If the user confirms the fall, the system sends the event to the backend, which notifies nearby responders. If a responder accepts, they receive navigation instructions; if no one responds in time, the system escalates to emergency services. This diagram visualizes the real-time logic and interactions behind a typical fall detection scenario.

Actor / Component Description
User Phone (App) The mobile device running the fall detection app, which communicates with sensors and backend.
Sensor Module Continuously reads accelerometer and gyroscope data to detect potential falls.
Fall Detection Module Classifies motion data and determines whether a fall has occurred.
User Interface Prompts user to confirm if a fall has occurred.
Backend Server Central server managing event lifecycles, responders, notifications, and escalation.
Notification Module Sends out alerts to responders and waits for acceptance.
Medical Responder App Receives event alerts, allows acceptance, and displays navigation.
Google Maps API Provides directions and AED locations to the responder.
Emergency Services (911) Fallback contact for cases where no responder accepts in a timely manner.
