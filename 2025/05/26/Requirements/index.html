1. Introduction


1.1 Purpose This Software Requirements Specification (SRS) describes the requirements for a smartphone-based fall detection application. The goal of this application is to provide faster emergency response and improve survival outcomes for individuals experiencing falls or sudden medical emergencies.

1.2 Document Conventions Requirements are tagged as "FR" for Functional Requirements and "NFR" for Non-Functional Requirements. Additional supporting elements are structured using appendices.

1.3 Intended Audience and Reading Suggestions This document is intended for software engineers, project managers, emergency services planners, and compliance officers. Readers should first read the Introduction and Overall Description sections before focusing on specific requirement categories.

1.4 Product Scope This application provides real-time fall detection using onboard smartphone sensors and notifies registered medical responders. Key features include medical profile registration, AED location mapping, and privacy-compliant alerting systems.

1.5 References

  • IEEE Std 830-1998

  • Google Maps API

  • Hellenic Mediterranean University – MobiFall Dataset

  • Japan’s Act on the Protection of Personal Information (APPI)

2. Overall Description


2.1 Product Perspective The fall detection app is a standalone product that integrates with various APIs and uses mobile sensors. It represents an original solution designed specifically to improve response times for fall-related emergencies.

2.2 Product Functions

  • Detect falls using smartphone sensors

  • Display countdown timer and initiate alerts

  • Notify nearby registered medical responders and emergency services

  • Allow registration of medical personnel

  • Show nearby AEDs and directions

2.3 User Classes and Characteristics

  • General Users: Elderly individuals, minimal tech experience

  • Medical Responders: Certified professionals, GPS tracking consent

  • Administrators: Manage backend systems and access controls

2.4 Operating Environment

  • OS: Android 10.0+, iOS 14.0+

  • Sensors: Accelerometer, Gyroscope, GPS

  • Connectivity: Wi-Fi/4G/5G for cloud sync and alerting

  • Storage: Minimum 50MB, Battery: ≥3000mAh

2.5 Design and Implementation Constraints

  • Must comply with APPI for user data handling

  • AES-256 encryption for all transmissions

  • Real-time performance: fall detection in <2s, alerts within <5s

2.6 User Documentation

  • User Manual

  • In-app onboarding and tooltips

  • FAQ section in app and online portal

2.7 Assumptions and Dependencies

  • Users allow access to required sensors and location

  • SMS fallback supported by the user’s carrier

  • MobiFall dataset remains accessible for testing

3. External Interface Requirements


3.1 User Interfaces

  • Elderly-friendly UI with large buttons and high-contrast text

  • Countdown timer after fall detection

  • Dual-language support: English and Japanese

3.2 Hardware Interfaces

  • Direct interaction with accelerometer, gyroscope, GPS module

3.3 Software Interfaces

  • Google Maps API for AED display and directions

  • Firebase for notifications and authentication

  • Machine learning engine for fall classification

3.4 Communications Interfaces

  • HTTPS and AES-256 encryption for secure transmission

  • Fallback to SMS if internet is unavailable

5. System Requirements (Nonfunctional Requirements)

  • NFR-1: The system shall keep a 99.9% uptime monthly (≤43.2 min downtime) (This is to ensure the app can help people 24 hours a day 7 days a week.)

  • NFR-2: The system shall have a fall detection latency <2s (to ensure help is dispatched as soon as possible.)

  • NFR-2.1: The system must alert dispatch within 5s of confirmation. (To ensure that help is on its way as soon as possible)

  • NFR-3: The system must be able to differentiate false-positive rate <5% across 1000 test cases (This is to not only build trust in the app but to also not waste emergency services time and money.)

  • NFR-3.1: The application should have a crash tolerance and recovery state(to ensure that there isn’t multiple messages sent out after one.)

  • NFR-4: The system must have bilingual support. (To ensure that the application can be used in a variety of countries).

  • NFR-4.1: The system must have an accessibility mode with font ≥18pt. (As said in our proposal, a majority of our use base will be the elderly who also suffer with impaired vision).

  • NFR-5: The system must have AES 128+ encryption (critical data AES-256). (To ensure data privacy).

  • NFR-5.1: The system must auto-delete location data within 1 hour. (To ensure data privacy but all to save server space).

5.1. System Requirements (Functional Requirements)

  • FR-1: The system will detect fall even using onboard sensors. (These sensors are widely available on mobile devices).

  • FR-1.1: The system will initiate a fall verification timer. (This is to ensure that if the person who fell is fine or they just dropped their device, they can cancel the message).

  • FR-2: The system will send an alert to the server. (To start the alert process).

  • FR-2.1: The system will send an alert to emergency personnel. (To inform them of the incident that occurred).

  • FR-2.2: The system will have a retry mechanism. (This is to ensure that all outcomes are true).

  • FR-3: The system will show nearby AEDs. (To inform users on the closest AED to their location).

  • FR-3.1: The system will display AED details. (This is to teach the AED user on how to properly apply and use the AED).

  • FR-4: They system will register Medical Personnel. (To ensure that people are not lying about qualifications).

  • FR-4.1: The system will get location data of medical personnel. (This is to always have their live location).

6.1 User Requirements (Functional requirements)

  • FR-1: Users expect the app to detect falls accurately using smartphone sensors such as accelerometers and gyroscopes, without the need for manual input. (This is so they can inform people about their fall without touching their phone).

  • FR-1.1: When a fall is detected, the system must notify nearby registered healthcare workers immediately, along with the user’s location and medical information. (To helppeople get the help they need).

  • FR-1.2: The app shall automatically alert emergency services with the fall location and victim’s condition when necessary. (To ensure that the lcoation of the fall is known to all parties involved).

Appendix A: Glossary


Term

Definition

Accelerometer

A smartphone sensor that measures motion and speed changes, such as when someone falls or moves suddenly.

AED (Automated External Defibrillator)

A portable medical device that can restart the heart by delivering an electric shock during sudden cardiac arrest.

API (Application Programming Interface)

A set of rules and tools that allow different software applications to communicate and work together.

APPI (Act on the Protection of Personal Information)

A Japanese law that regulates how organizations handle personal data to protect user privacy.

Crash Tolerance

The app’s ability to recover and resume normal operation even after an unexpected shutdown or failure.

Countdown Timer

A visual timer used in the app to let users cancel an emergency alert if they are not actually injured.

CPR (Cardiopulmonary Resuscitation)

An emergency technique used when someone’s heart or breathing stops, often performed by trained bystanders.

DNR Status (Do Not Resuscitate)

A medical instruction indicating that a person does not want CPR or other life-saving treatments if their heart stops.

Fall Detection

The ability of the app to recognize when a person has fallen by analysing sensor data from the phone.

False Positive

An incorrect detection where the system thinks a fall occurred, but the user has not fallen.

Gyroscope

A smartphone sensor that measures the rotation or tilt of the device, often used alongside accelerometers for better fall detection.

Healthcare Worker

A medical professional (like a doctor or nurse) who can register in the app to respond to emergencies.

Latency

The time delay between detecting a fall and performing the required response actions.

Medical Profile

A user’s personal health information, including age, medical history, allergies, and emergency preferences.

Medical Personnel Verification

A process where the app checks if a person is a real healthcare worker using ID or credentials.

MobiFall Dataset

A publicly available dataset used to test and improve fall detection algorithms.

Monitoring System

A system used by developers or administrators to check if the app is working properly or if any errors occur.

NFR

Non-Functional Requirement.

Retry Mechanism

A system feature that tries several times to send an emergency alert if it fails the first time.

Sigma

The internal name or version label of the fall detection system or app.

Smartphone Sensors

Built-in tools like accelerometers and gyroscopes that help detect movement, orientation, and falls.

System Availability (Uptime)

The percentage of time the app is active and functioning properly, such as 99.9% uptime per month.

Volunteer Responder

A trained person (such as someone CPR-certified) who is not a healthcare worker but can assist in emergencies.
